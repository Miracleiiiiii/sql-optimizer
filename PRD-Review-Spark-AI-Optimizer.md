# PRD 评审: Spark AI Optimizer 可实施性评估

## 1. 评审结论

该 PRD 可实施，建议立项推进。

但产品形态需要从“AI 自动调参系统”收敛为“Spark 任务诊断与参数推荐系统”。第一阶段只做诊断、解释、建议和人工采纳，不做生产参数自动修改。这样既能快速落地，又能降低大模型幻觉、误调生产任务和权限改造带来的风险。

## 2. 可实施性判断

| 维度 | 结论 | 说明 |
| --- | --- | --- |
| 数据可获得性 | 可实施 | 线下 Spark History Server、YARN ResourceManager 已验证可访问，Event Log 已开启 |
| 技术复杂度 | 中等偏高 | 指标采集和规则诊断可控，难点在跨版本兼容、指标缺失和推荐准确性 |
| AI 适配度 | 可实施 | 大模型适合做归因解释和报告生成，不适合直接做底层指标计算 |
| 业务价值 | 明确 | 慢任务诊断、资源浪费识别和调参建议都有直接价值 |
| 生产风险 | 可控 | MVP 不自动改参，所有建议人工确认 |
| 投入周期 | 合理 | 技术验证 1-2 周，MVP 3-5 周可完成 |

## 2.1 已确认实施条件

| 项目 | 结果 |
| --- | --- |
| Spark 版本 | 线下 3.3.1，线上 3.3.2 |
| History Server | `http://hadoop102:18080` 可访问 |
| History Server API | jobs、stages、executors、environment、sql 已验证可返回 |
| YARN API | `http://hadoop102:8088` 可访问 |
| Event Log | 已开启，目录 `hdfs://hadoop101:8020/spark-history` |
| Event Log 保留周期 | 7 天 |
| MVP 任务类型 | 独立 `spark-submit`，按 applicationId 分析 |

| 调度平台 | MVP 暂不接入，由用户传入 applicationId |
| 大模型 | MVP 先使用外部 OpenAI API，模型配置为 GPT-5.4 或 GPT-5.5 |
| SQL 入模 | 允许 SQL 文本进入外部模型，保留脱敏和审计能力 |

## 3. 关键评审意见

### 3.1 不建议直接读取 Spark UI HTML

Spark UI 页面结构可能随版本变化，直接解析 HTML 稳定性较差。建议优先使用：

- Spark History Server REST API。
- Spark Event Log。
- YARN ResourceManager API。
- Hive Metastore 或离线表画像。

Spark UI 可以作为人工跳转入口，而不是核心采集依赖。

### 3.2 大模型定位需要清晰

大模型不应负责：

- 直接解析原始 UI 页面。
- 计算 task、stage、executor 指标。
- 无约束地生成参数值。
- 自动修改生产调度参数。

大模型适合负责：

- 将结构化诊断结果整理成自然语言报告。
- 解释多个异常指标之间的因果关系。
- 生成带证据和风险的参数建议。
- 输出验证方案。

MVP 使用外部 OpenAI API，模型名称通过配置项指定为 GPT-5.4 或 GPT-5.5。模型调用层需要抽象 provider 接口，后续如果切换私有化模型，不影响指标采集和规则诊断主链路。

### 3.3 必须有规则引擎兜底

建议采用“规则引擎 + 大模型”的双层结构：

```text
原始指标 -> 标准化指标 -> 规则诊断 -> LLM 解释增强 -> 参数建议报告
```

规则引擎负责确定事实边界，大模型负责表达、归因和组合建议。这样可以降低模型幻觉风险。

### 3.4 MVP 范围需要更小

原 PRD 同时覆盖 Event Log、Hive、批量巡检、闭环、半自动调参，范围偏大。建议 MVP 只做：

- 输入独立 `spark-submit` 任务的 applicationId。
- 采集 Spark History Server 和 YARN 数据。
- 抽取应用、stage、executor、SQL 基础指标。
- 诊断 GC、shuffle spill、数据倾斜、并行度不足。
- 输出 4 个核心参数建议。
- 保存报告。

Event Log、Hive 元数据、批量巡检和闭环放到后续阶段。

### 3.5 参数推荐必须是“建议范围”，不是绝对答案

Spark 参数调优受以下因素影响：

- 集群资源上限。
- YARN 队列容量。
- 作业输入数据量。
- 上游分区和文件大小。
- 当前任务是否和其他任务并发运行。
- Spark 版本和动态资源配置。

因此建议输出推荐范围和验证方式，例如：

- 推荐将 `spark.sql.shuffle.partitions` 从 200 调整到 600-1000。
- 推荐先灰度验证 1 次。
- 观察总耗时、p95 task duration、spill 和 GC ratio。

## 4. 主要风险

| 风险 | 严重程度 | 说明 | 应对 |
| --- | --- | --- | --- |
| 指标缺失 | 高 | History Server API 不一定包含全部 task 明细 | P1 增加 Event Log 解析 |
| Spark 版本差异 | 中 | 不同版本 API 字段可能不同 | 采集层做 adapter |
| 推荐不准确 | 高 | 调参依赖上下文，单次任务信息不足 | 输出推荐范围、置信度和验证方式 |
| 大模型幻觉 | 高 | 可能生成无依据建议 | JSON schema、规则兜底、证据约束 |
| SQL 泄露 | 高 | SQL 中可能包含敏感表名和字段 | 脱敏、截断、审计、权限控制 |
| Hive 统计不准 | 中 | 表统计可能过期 | 作为辅助证据，标注可信度 |
| 自动改参风险 | 高 | 错误参数可能影响生产链路 | MVP 禁止自动应用 |
| Event Log 过期 | 中 | 当前保留周期 7 天，超过周期后无法补充解析 | MVP 先使用 History Server API，P1 增加快照落库 |

## 5. 推荐实施路径

### 5.1 P0 技术验证

目标：验证数据源可用性。

输出：

- 给定 applicationId，可以拉取 History Server 数据。
- 生成统一指标 JSON。
- 证明可以识别至少 2 类典型问题。

周期：1-2 周。

### 5.2 P1 MVP

目标：完成单任务诊断闭环。

输出：

- 单任务分析 API。
- 规则诊断引擎。
- AI 报告生成。
- 参数推荐表。
- 报告存储和查询。

周期：3-5 周。

### 5.3 P2 批量巡检

目标：从单点分析扩展到平台治理。

输出：

- 每日任务扫描。
- 异常任务榜单。
- 慢任务和资源浪费任务识别。

周期：3-4 周。

### 5.4 P3 优化闭环

目标：验证推荐是否真的有效。

输出：

- 推荐采纳记录。
- 优化前后对比。
- 同任务历史基线。
- 推荐有效率统计。

周期：4-6 周。

## 6. 是否需要调整 PRD

需要。已建议将 PRD 调整为：

- 明确 MVP 不做自动改参。
- 明确大模型只做解释增强和报告生成。
- 明确规则引擎是诊断主干。
- 明确 Event Log、Hive、批量巡检作为后续增强。
- 明确验收指标分 MVP 和完整版。
- 增加安全、脱敏、权限和输出校验要求。

## 7. 最终建议

建议进入 P0 技术验证。优先验证以下问题：

- 线上 Spark History Server、YARN API、Hive Metastore thrift 地址是否可通过配置接入。
- 是否有可复核的慢任务样本用于验证诊断准确率。
- 外部 OpenAI API 的网络访问、密钥管理、模型配置和调用额度。
- SQL 虽允许进入外部模型，但仍需确认是否需要表名、字段名、库名脱敏。

只要以上数据源能打通，MVP 就具备明确可实施性。
